# ТЗ для разработки: сервис «Лента вдохновения» Gloria Jeans

## 0. Главный продуктовый принцип

Это не каталог и не PDP. Это визуальный mobile-first discovery-сервис.

Главный сценарий:

1. Пользователь открывает плотную визуальную ленту изображений.
2. Скроллит ее как нативный doomscroll-сервис.
3. Тапает по понравившемуся товару.
4. Видит детальный экран товара.
5. Видит похожие товары.
6. Бесшовно продолжает исходную ленту с того места, где остановился.

Нельзя ломать поток:

- модальными окнами;
- фильтрами на ленте;
- ценниками на плитках;
- названиями на плитках;
- кнопками корзины или избранного на плитках;
- потерей позиции скролла;
- перезапуском ленты после возврата назад;
- классическим e-commerce PDP как основным сценарием.

## 1. Цель документа

Документ задает явное техническое задание для разработчика v1 сервиса «Лента вдохновения» для Gloria Jeans.

Разработчик должен на основании этого ТЗ:

- реализовать mobile-first веб-сервис бесконечной визуальной ленты;
- подключить товарный фид;
- нормализовать данные товаров и категорий;
- реализовать API для ленты, detail, similar и продолжения ленты;
- обеспечить сохранение состояния ленты, cursor/anchor и возврат назад;
- заложить graceful degradation, аналитику и инфраструктурные требования.

## 2. Термины

| Термин | Значение |
| --- | --- |
| Лента | Бесконечная визуальная сетка изображений товаров без коммерческого шума |
| Плитка | Один элемент сетки ленты, содержит только изображение |
| Detail | Детальный экран товара внутри discovery-сценария |
| Similar | Блок похожих товаров под открытым товаром |
| Continue feed | Продолжение исходной ленты после блока similar |
| Cursor | Серверный указатель на следующую порцию ленты |
| Anchor | Товар и позиция, из которой пользователь открыл detail |
| Section | Верхний раздел: Женщины, Мужчины, Дети, Подростки |
| Feed version | Версия импортированного фида |
| Seen set | Набор товаров, уже показанных пользователю в текущем потоке |

## 3. Контекст продукта

Нужно разработать mobile-first веб-сервис в формате бесконечной визуальной ленты товаров.

Ориентиры:

- по поведению: нативное мобильное приложение;
- по потреблению контента: doomscroll;
- по визуалу: плотная сетка как Pinterest;
- по назначению: вдохновение и discovery, а не каталог.

Пользовательский сценарий:

> «Я не знаю точно, что ищу, но хочу посмотреть, что нравится».

Сервис должен помогать пользователю:

- вдохновиться;
- быстро просмотреть много визуального контента;
- открыть понравившийся товар;
- увидеть похожие товары;
- продолжить исходную ленту без потери контекста.

## 4. Что не входит в v1

В v1 не реализуем:

- полноценный интернет-магазин;
- корзину;
- фильтры на ленте;
- сортировки на ленте;
- сложные модальные сценарии;
- отдельный PDP в классическом виде;
- outfit builder;
- ML-рекомендации;
- полноценный SEO-каталог;
- ручной мерчандайзинг всей ленты;
- авторизацию, если она не требуется для аналитики или deep link.

Сервис не должен постепенно превратиться в обычный каталог с другим гридом.

## 5. Целевая платформа и UX-приоритеты

Основная платформа: мобильный веб.

Desktop допускается как адаптация того же сценария, но не как отдельный продукт.

UX-приоритеты в порядке важности:

1. Быстрый первый экран.
2. Плавный вертикальный скролл.
3. Минимум текста.
4. Минимум решений от пользователя.
5. Быстрый возврат в тот же контекст.
6. Отсутствие ощущения страниц и модалок.
7. Отсутствие коммерческого шума в ленте.

## 6. Информационная архитектура

### 6.1. Верхние разделы

Единственное явное разбиение верхнего уровня:

| Раздел | Содержание |
| --- | --- |
| Женщины | Женский ассортимент |
| Мужчины | Мужской ассортимент |
| Дети | Мальчики и девочки |
| Подростки | Мальчики-подростки и девочки-подростки |

Для «Дети» и «Подростки» можно использовать вложенные вкладки:

- Дети: мальчики, девочки;
- Подростки: мальчики-подростки, девочки-подростки.

Требования:

- переключение раздела не должно ощущаться как переход в другой продукт;
- при смене раздела нужно сохранять независимое состояние каждой ленты;
- если пользователь вернулся в предыдущий раздел, нужно восстановить его cursor, scroll position и уже загруженные items;
- визуальная модель остается единой: это всегда лента изображений.

### 6.2. Роутинг

Рекомендуемая структура роутов:

| Route | Назначение |
| --- | --- |
| `/` | Редирект на дефолтный раздел или последний открытый раздел |
| `/feed/:section` | Лента верхнего раздела |
| `/feed/:section/:subsection` | Лента вложенного раздела, если нужен |
| `/product/:productId` | Detail товара внутри discovery-сценария |

Допустимо использовать query-параметры для восстановления состояния:

- `cursor`;
- `anchor_product_id`;
- `position_index`;
- `feed_version`;
- `session_id`.

Важно: URL может быть shareable, но восстановление точного scroll state из URL не является обязательным для v1. Для back-навигации состояние должно восстанавливаться обязательно.

## 7. Главный экран ленты

### 7.1. Общий вид

Лента состоит только из изображений товаров.

На плитке запрещены:

- название;
- цена;
- старая цена;
- скидка;
- sale/new/bestseller шильдики;
- рейтинг;
- кнопка избранного;
- кнопка корзины;
- промо-тексты;
- hover-кнопки на desktop как основной сценарий.

Плитка является визуальным крючком, а не e-commerce карточкой.

### 7.2. Mobile-сетка

Требования:

- сетка занимает всю ширину экрана;
- базовый вариант: 2 колонки;
- допустимо A/B или feature flag для 3 колонок, если изображения остаются читаемыми;
- отступы между плитками минимальные;
- нет классических карточных контейнеров;
- нет теней, рамок, «карточной» подложки;
- плитки могут иметь разную высоту, если известен aspect ratio;
- при неизвестном aspect ratio использовать заранее заданный ratio, чтобы избежать layout shift.

Рекомендуемые параметры для v1:

- `grid-template-columns: repeat(2, minmax(0, 1fr))`;
- gap: 2-4 px;
- padding page: 0-4 px;
- image object-fit: cover;
- image width: 100%;
- image loading: lazy для всего ниже первого экрана.

### 7.3. Desktop-сетка

Требования:

- 4-6 колонок в зависимости от ширины;
- сохраняется плотный визуальный характер;
- не превращать desktop в каталог;
- не добавлять фильтры, боковые панели и карточные подписи.

### 7.4. Визуальный ритм

Лента не должна выглядеть как случайная выгрузка фотографий из каталога.

Нужно предусмотреть правила микса:

- не показывать подряд много одинаковых предметных фото;
- чередовать модельные фото, предметку, крупные планы, если такие типы есть в данных;
- не показывать рядом дубли одного товара в разных цветах, если они визуально почти одинаковы;
- не допускать стены одинаковых футболок, джинсов или базовых вещей;
- не показывать товары без качественного главного фото;
- избегать визуальных провалов: пустой фон, обрезанные фото, технические изображения.

Если тип фото в фиде не размечен, для v1 использовать эвристику:

- брать первое или главное изображение товара;
- исключать товары без валидного изображения;
- исключать повтор одного `external_product_id`;
- ограничивать количество подряд идущих товаров одной `leaf_category`;
- ограничивать количество подряд идущих товаров одного `color`, если цвет известен;
- заложить поля `photo_type` и `image_quality_score` в модель данных как nullable для будущего развития.

## 8. Detail-экран товара

### 8.1. Поведение при тапе

При тапе по плитке открывается detail view товара в том же discovery-сценарии.

Допустимые варианты:

- переход на отдельный route с сохранением состояния ленты;
- нативноподобный transition;
- detail view как самостоятельный экран, из которого back возвращает в исходную позицию.

Недопустимо:

- маленькая модалка поверх ленты;
- bottom sheet как основной сценарий;
- тяжелая классическая PDP-страница;
- потеря позиции пользователя в ленте;
- принудительное открытие внешнего сайта без промежуточного detail в сервисе.

### 8.2. Содержимое detail

Detail должен содержать:

1. Hero-фото товара на всю ширину экрана.
2. Дополнительные фото, если доступны.
3. Название.
4. Цена.
5. Старая цена, если есть.
6. CTA перехода в товар, на сайт или в приложение, если нужен в v1.
7. Блок похожих товаров.
8. Продолжение исходной ленты после похожих.

Коммерческая информация появляется только здесь, не на плитке ленты.

### 8.3. Detail не должен становиться классическим PDP

В v1 не добавлять:

- подробный размерный блок, если он ведет к корзине;
- сложный выбор SKU;
- sticky cart CTA;
- cross-sell-блоки каталожного типа;
- SEO-тексты;
- характеристики на несколько экранов;
- отзывы и рейтинги.

Если бизнесу нужен переход к покупке, использовать один аккуратный CTA:

- «Перейти к товару»;
- «Открыть на сайте»;
- «Открыть в приложении».

## 9. Ключевой UX-паттерн: opened product -> similar -> continue feed

После открытия товара пользователь должен видеть единый вертикальный поток:

```text
Открытый товар
↓
Похожие товары
↓
Продолжение исходной ленты с позиции N+1
```

Пользователь не должен чувствовать, что его выкинуло из ленты.

Критически важно:

- если пользователь открыл товар на позиции `N`, после similar он должен увидеть товары с позиции `N + 1`;
- нельзя начинать ленту заново;
- нельзя подмешивать случайную новую ленту без учета anchor;
- нельзя терять уже просмотренные товары;
- similar не должен заменять исходную ленту, он только вставляется между открытым товаром и продолжением.

## 10. Состояние ленты

### 10.1. Клиентское состояние

Клиент должен хранить состояние отдельно для каждого раздела и subsection:

```ts
type FeedState = {
  section: RootSection;
  subsection?: string;
  items: FeedItem[];
  cursor: string | null;
  feedVersion: string | null;
  scrollY: number;
  positionIndexByProductId: Record<string, number>;
  alreadySeenProductIds: string[];
  openedProductId?: string;
  anchorProductId?: string;
  anchorPositionIndex?: number;
  sessionId: string;
  isLoading: boolean;
  hasMore: boolean;
};
```

Требования:

- хранить `items`, а не только cursor;
- хранить `scrollY` перед переходом в detail;
- хранить `positionIndexByProductId`;
- хранить `alreadySeenProductIds` для антидублей;
- состояние не сбрасывается при back;
- состояние не сбрасывается при переключении между detail и feed;
- состояние разделов изолировано друг от друга.

### 10.2. Anchor-состояние

При открытии товара клиент фиксирует:

```ts
type FeedAnchor = {
  section: RootSection;
  subsection?: string;
  cursor: string | null;
  anchorProductId: string;
  positionIndex: number;
  alreadySeenProductIds: string[];
  openedProductId: string;
  feedVersion: string | null;
  sessionId: string;
};
```

Anchor используется для:

- получения continuation после similar;
- восстановления back-навигации;
- аналитики detail open;
- предотвращения дублей.

### 10.3. Back-навигация

При возврате из detail в ленту:

- пользователь возвращается на ту же позицию;
- scroll position восстанавливается;
- лента не запрашивается заново с начала;
- уже загруженные изображения не прыгают;
- выбранный section и subsection сохраняются;
- cursor сохраняется;
- items сохраняются;
- уже просмотренные товары не исчезают и не перемешиваются.

Рекомендуемая реализация:

- использовать History API state для anchor и scroll;
- хранить feed store выше уровня route;
- восстанавливать scroll после отрисовки сохраненных items;
- использовать stable key `product.id` для плиток;
- задавать изображениям известные размеры или aspect ratio.

## 11. Similar: похожие товары

### 11.1. Назначение

Similar является главным механизмом discovery после открытия товара.

Похожие товары должны быть ранжированы по степени похожести, а не просто выбраны из той же категории.

### 11.2. Сигналы похожести

Приоритет сигналов:

1. Та же leaf-категория.
2. Цвет.
3. Тип товара.
4. Пол или возрастная ветка.
5. Силуэт, фасон, fit, если есть в атрибутах.
6. Сезон или коллекция, если есть.
7. Цена или ценовой диапазон.
8. Популярность.
9. Наличие.

### 11.3. Каскад fallback

Если точных похожих мало, алгоритм должен расширять круг:

1. Та же `leaf_category` + тот же цвет + совпадающие параметры.
2. Та же `leaf_category`.
3. Та же родительская категория.
4. Та же верхняя ветка, например `Женщины -> Одежда`.
5. Популярные товары из этой ветки.
6. Продолжение исходной ленты.

Блок similar не должен быть пустым. Если похожих недостаточно, нужно использовать graceful fallback и логировать `similar_fallback_used`.

### 11.4. Скоринговая модель для v1

Реализовать простую интерпретируемую scoring-функцию:

```text
score =
  100, если совпадает leaf_category
  + 30, если совпадает color
  + 25, если совпадает product_type
  + 20, если совпадает gender_or_age_group
  + 15, если совпадает fit/silhouette
  + 10, если совпадает season/collection
  + 10, если price в диапазоне ±25%
  + popularity_score_normalized
```

Правила:

- сначала фильтровать запрещенные товары;
- потом считать score;
- потом сортировать по score;
- при равном score использовать popularity или стабильный deterministic seed;
- результат должен быть стабильным в рамках одной feed version.

### 11.5. Антидубли в similar

В similar нельзя показывать:

- текущий товар;
- товары, уже показанные выше в этом же similar-блоке;
- дубли одного SKU;
- один и тот же товар в разных местах одной выдачи;
- товары без фото;
- товары не в продаже;
- товары не из выбранной возрастно-половой ветки, если это нарушает ожидание пользователя;
- товары из `exclude_product_ids`.

## 12. Продолжение основной ленты после similar

### 12.1. Требование

После блока similar нужно продолжить исходную ленту.

Клиент и API должны использовать:

- `section`;
- `subsection`;
- `cursor`;
- `anchor_product_id`;
- `position_index`;
- `already_seen_product_ids`;
- `opened_product_id`;
- `feed_version`;
- `session_id`.

### 12.2. Поведение

Если пользователь открыл товар на позиции `N`, continuation должен вернуть товары с позиции `N + 1`.

Endpoint continuation не должен:

- начинать выдачу с начала;
- игнорировать anchor;
- возвращать opened product;
- возвращать товары из `already_seen_product_ids`;
- ломать порядок текущей feed version.

### 12.3. Стабильность ленты

Нельзя, чтобы при каждом обновлении фида лента полностью перемешивалась для пользователя в текущей сессии.

Требования:

- feed order должен быть стабильным в рамках `feed_version`;
- session может закрепляться за feed_version на время жизни сессии;
- при обновлении фида текущая активная сессия продолжает использовать старую version, если это технически возможно;
- если старая version недоступна, использовать graceful migration без сброса UX.

## 13. Данные и фид

### 13.1. Источник данных

Источник данных v1:

```text
https://storage-cdn11.gloria-jeans.ru/catalog/feeds/auto-merch/auto-merch-feed-cc-central.xml
```

Формат источника нужно изолировать от фронта.

Требования:

- фронт не парсит XML/YML напрямую;
- backend/importer нормализует данные;
- API отдает стабильный JSON;
- формат источника можно заменить без переписывания фронта.

### 13.2. Нормализованная модель товара

Минимальная модель:

```ts
type Product = {
  id: string;
  externalProductId: string;
  name: string;
  price: number | null;
  oldPrice: number | null;
  currency: string;
  categoryId: string | null;
  categoryPath: string[];
  rootSection: "women" | "men" | "kids" | "teens" | "unknown";
  genderOrAgeGroup: string | null;

  color: string | null;
  normalizedColor: string | null;

  fit: string | null;

  season: string | null;

  mainImageUrl: string | null;
```

### 13.3. Модель категории

```ts
type CategoryNode = {
  id: string;
  name: string;
  parentId: string | null;
  path: string[];
  level: number;
  rootSection: Product["rootSection"];
};
```

Нужно восстановить полный путь категории, даже если товар лежит на 5-6 уровне.

Пример:

```text
Женщины -> Одежда -> Джинсы -> Wide leg -> Голубые
```

Сервис должен уметь определить:

- `root_section = Женщины`;
- `level_2 = Одежда`;
- `level_3 = Джинсы`;
- `leaf_category = Голубые`.

Это используется для:

- выдачи ленты верхнего уровня;
- similar;
- fallback-логики;
- аналитики;
- quarantine битых записей.

### 13.4. Политика битых данных

| Ситуация | Поведение v1 |
| --- | --- |
| Нет фото | Не показывать в ленте и similar |
| Нет цены | Можно показывать в ленте, но в detail скрыть цену или показывать «Цена недоступна» по решению продукта |
| Нет категории | Отправить в quarantine, не публиковать |
| Нет пола или возрастной ветки | Определить по `category_path`; если не удалось, quarantine |
| Товар исчез из фида | Скрыть после обновления |
| Товар не в наличии | Не показывать в ленте v1 |
| Некорректный product_url | Не показывать CTA, логировать ошибку |
| Битое изображение | Исключить при импорте, если проверяется; иначе скрыть плитку на клиенте при ошибке |

## 14. Импорт и обновление данных

### 14.1. Import pipeline

Рекомендуемый pipeline:

```text
download feed
-> validate raw feed
-> parse
-> normalize products
-> rebuild category tree
-> classify root_section
-> validate images and required fields
-> deduplicate
-> score or flag quality
-> write new feed_version
-> publish version atomically
-> hide products missing from new feed
-> write import logs
```

### 14.2. Требования к импорту

Нужно предусмотреть:

- регулярную загрузку фида;
- идемпотентность по `external_product_id`;
- версионирование записи;
- `updated_at`;
- `feed_version`;
- политику скрытия товаров, отсутствующих в новом фиде;
- лог ошибок импорта;
- ручной перезапуск импорта;
- сохранение последней успешной версии фида;
- fallback на последнюю успешную версию, если текущий импорт упал.

### 14.3. Публикация версии

Новая версия фида должна публиковаться атомарно:

- пока импорт не завершен успешно, API отдает старую успешную версию;
- при ошибке импорта старая версия не ломается;
- после успешного импорта новый `feed_version` становится доступен новым сессиям;
- активные сессии по возможности продолжают старую версию.

## 15. API-контракты

### 15.1. Общие требования

Все API должны возвращать JSON.

Общие поля ошибок:

```json
{
  "error": {
    "code": "string",
    "message": "string",
    "request_id": "string"
  }
}
```

Требования:

- API не должен отдавать товары без фото;
- API не должен отдавать товары не в продаже;
- API должен поддерживать `exclude_product_ids`;
- порядок выдачи должен быть стабильным в рамках `feed_version`;
- `limit` должен иметь server-side cap;
- cursor должен быть opaque string, клиент не должен его разбирать.

### 15.2. GET `/api/feed`

Назначение: получить первую или следующую порцию ленты.

Query-параметры:

| Параметр | Тип | Обязателен | Описание |
| --- | --- | --- | --- |
| `section` | string | Да | `women`, `men`, `kids`, `teens` |
| `subsection` | string | Нет | Например `boys`, `girls`, `teen_boys`, `teen_girls` |
| `cursor` | string | Нет | Opaque cursor следующей страницы |
| `limit` | number | Нет | Рекомендуемо 30-60, server cap обязателен |
| `exclude_product_ids` | string | Нет | CSV или повторяющийся query-param |
| `session_id` | string | Да | ID пользовательской сессии |
| `feed_version` | string | Нет | Версия фида для стабильной сессии |

Ответ:

```json
{
  "items": [
    {
      "id": "string",
      "main_image_url": "string",
      "image_aspect_ratio": 0.75,
      "position_index": 0
    }
  ],
  "next_cursor": "string",
  "feed_version": "2026-04-20T10:00:00Z",
  "has_more": true
}
```

В `items` ленты запрещено отдавать поля цены и названия, если фронт может случайно их показать. Detail получает коммерческие поля через отдельный endpoint.

### 15.3. GET `/api/products/{product_id}`

Назначение: получить detail товара.

Ответ:

```json
{
  "id": "string",
  "name": "Джинсы wide leg",
  "price": 1299,
  "old_price": 1999,
  "currency": "RUB",
  "images": [
    {
      "url": "string",
      "aspect_ratio": 0.75
    }
  ],
  "category_path": ["Женщины", "Одежда", "Джинсы"],
  "product_url": "https://example.com/product",
  "availability_status": "in_stock"
}
```

Если товар стал недоступен:

- вернуть `404` или `410` с понятным code;
- клиент показывает мягкое сообщение;
- клиент все равно может показать similar, если backend позволяет получить fallback по последнему известному товару.

### 15.4. GET `/api/products/{product_id}/similar`

Назначение: получить похожие товары для detail.

Query-параметры:

| Параметр | Тип | Обязателен | Описание |
| --- | --- | --- | --- |
| `section` | string | Да | Текущий root section |
| `subsection` | string | Нет | Текущий subsection |
| `limit` | number | Нет | Рекомендуемо 12-24 |
| `exclude_product_ids` | string | Нет | Товары, которые уже нельзя показывать |
| `session_id` | string | Да | ID сессии |
| `feed_version` | string | Нет | Версия фида |

Ответ:

```json
{
  "items": [
    {
      "id": "string",
      "main_image_url": "string",
      "image_aspect_ratio": 0.75,
      "score": 155
    }
  ],
  "strategy": "same_category_color_then_parent_fallback",
  "fallback_used": false
}
```

Поле `strategy` нужно для отладки и аналитики. Пользователю его не показываем.

### 15.5. GET `/api/feed/continue`

Назначение: вернуть хвост исходной ленты после позиции, на которой пользователь открыл товар.

Query-параметры:

| Параметр | Тип | Обязателен | Описание |
| --- | --- | --- | --- |
| `section` | string | Да | Текущий root section |
| `subsection` | string | Нет | Текущий subsection |
| `cursor` | string | Нет | Cursor исходной ленты |
| `anchor_product_id` | string | Да | Товар, с которого открыт detail |
| `position_index` | number | Да | Позиция anchor в исходной ленте |
| `exclude_product_ids` | string | Нет | Уже показанные товары |
| `limit` | number | Нет | Рекомендуемо 30-60 |
| `session_id` | string | Да | ID сессии |
| `feed_version` | string | Нет | Версия фида |

Ответ:

```json
{
  "items": [
    {
      "id": "string",
      "main_image_url": "string",
      "image_aspect_ratio": 0.75,
      "position_index": 101
    }
  ],
  "next_cursor": "string",
  "feed_version": "2026-04-20T10:00:00Z",
  "has_more": true
}
```

## 16. Алгоритм выдачи ленты

### 16.1. Базовая фильтрация

Перед ранжированием исключить:

- товары без `main_image_url`;
- товары с `availability_status != in_stock`;
- товары с `publication_status != published`;
- товары без root section;
- товары из quarantine;
- товары из `exclude_product_ids`;
- дубли `external_product_id`;
- варианты, которые визуально являются дублями, если есть family/group id.

### 16.2. Формирование порядка

Для v1 допустим deterministic ranking:

```text
eligible products
-> group by category / product_type / color
-> apply visual rhythm rules
-> sort with stable seed by feed_version + section + session bucket
-> paginate by cursor
```

Требования:

- порядок стабилен внутри `feed_version`;
- обновление фида не должно перемешивать текущую пользовательскую сессию;
- cursor не должен зависеть от offset в нестабильной выборке без version;
- при догрузке не должно быть дублей.

### 16.3. Правила visual rhythm для v1

Минимальные правила:

- не более 2 товаров подряд одной `leaf_category`;
- не более 2 товаров подряд одного `product_type`, если поле доступно;
- не более 2 товаров подряд одного `normalizedColor`, если поле доступно;
- не показывать рядом товары одной product family, если есть group id;
- если нет нужных атрибутов, применять хотя бы дедупликацию и category shuffle.

## 17. Производительность

### 17.1. Первый экран

Требования:

- первый экран должен появляться быстро;
- изображения первого экрана грузить приоритетно;
- изображения ниже первого экрана грузить lazy;
- skeleton или placeholder допустим, но он не должен выглядеть как тяжелый e-commerce;
- не блокировать первый экран ожиданием similar;
- не блокировать первый экран аналитикой;
- не блокировать первый экран полной загрузкой фида;
- не грузить оригинальные тяжелые изображения из фида напрямую.

Рекомендуемые практики:

- `fetchpriority="high"` для первых 4-6 изображений;
- responsive image sizes;
- CDN или image proxy;
- заранее известный aspect ratio;
- placeholder через blur или dominant color, если возможно.

### 17.2. Скролл

Требования:

- плавный вертикальный скролл;
- предзагрузка следующей порции до конца списка;
- отсутствие layout shift при загрузке изображений;
- контроль памяти в длинных doomscroll-сессиях;
- стабильные ключи элементов;
- отсутствие синхронных тяжелых операций во время scroll.

Рекомендуемые практики:

- виртуализация или windowing для длинного списка;
- IntersectionObserver для lazy loading и impressions;
- requestIdleCallback или batched analytics;
- throttling scroll-depth analytics;
- хранение размеров плиток.

### 17.3. Изображения

Требования:

- использовать адаптивные размеры;
- отдавать мобильному экрану мобильное разрешение;
- не тянуть оригиналы из фида напрямую, если они тяжелые;
- поддержать CDN/image proxy, если доступно;
- fallback placeholder при ошибке загрузки;
- скрывать плитку, если изображение окончательно не загрузилось и это не ломает сетку.

## 18. Слабая сеть и ошибки

| Ситуация | Поведение |
| --- | --- |
| Лента не загрузилась | Короткое сообщение и кнопка «Повторить» |
| Similar не загрузились | Показать продолжение основной ленты |
| Изображение не загрузилось | Placeholder или скрыть плитку |
| Товар стал недоступен | Мягкое сообщение и похожие товары |
| Фид временно недоступен | Использовать последнюю успешную версию, если возможно |
| Пустой раздел | Fallback на общий популярный поток того же root section или мягкое empty state |
| API вернул дубли | Клиент дополнительно фильтрует по `id` |
| Истек cursor | Запросить feed по anchor или показать continuation fallback |

Все ошибки должны логироваться, но не блокировать интерфейс.

## 25. Критерии приемки

### 25.1. Лента

- На мобильной ширине отображается плотная визуальная сетка.
- В плитках только изображения.
- Нет цен, названий, шильдиков и кнопок.
- Лента занимает всю ширину экрана.
- Работает lazy loading.
- Скролл плавный.
- Есть разделы Женщины, Мужчины, Дети, Подростки.
- Переключение раздела не превращает сервис в каталог.

### 25.2. Открытие товара

- Тап по плитке открывает detail.
- Detail не является модалкой.
- Hero-фото занимает всю ширину экрана.
- Название и цена отображаются только в detail.
- Ниже отображается блок похожих товаров.
- После похожих продолжается исходная лента.
- Back возвращает пользователя в прежнюю позицию.

### 25.3. Similar

- Похожие товары ранжируются по категории, цвету и параметрам.
- При отсутствии точных похожих работает fallback.
- Нет дублей.
- Не показывается текущий товар.
- Не показываются товары без фото.
- Не показываются недоступные товары.
- Возвращается `strategy` для отладки.

### 25.4. Данные

- Фид парсится.
- Категорийное дерево восстановлено.
- Для товара определяется верхняя ветка.
- Описана и реализована политика битых записей.
- Обновления фида идемпотентны.
- Есть feed version.
- При падении импорта сохраняется последняя успешная версия.

### 25.5. API

Документированы и реализованы:

- лента;
- карточка товара;
- similar;
- продолжение ленты;
- cursor;
- anchor;
- feed version;
- exclude ids.

## 27. Риски

| Риск | Последствие | Митигация |
| --- | --- | --- |
| Нет нормальных атрибутов цвета/fit | Similar будет слабее | Использовать category fallback и подготовить nullable-поля |
| Тяжелые изображения из фида | Медленный первый экран | CDN/image proxy, responsive sizes |
| Нет стабильного feed version | Лента прыгает между обновлениями | Ввести feed_version и session binding |
| Back не восстанавливает scroll | UX ощущается сломанным | Store выше route, History state, stable keys |
| Similar пустой | Discovery обрывается | Fallback cascade |
| Много дублей SKU/цветов | Лента выглядит как выгрузка каталога | Dedup + visual rhythm rules |
| Бизнес попросит цены на плитке | Продукт станет каталогом | Зафиксировать запрет в acceptance criteria |

## 28. Минимальный результат v1

Минимально успешная v1 считается готовой, если:

- пользователь открывает mobile-first ленту;
- видит только изображения товаров;
- быстро скроллит без заметных прыжков;
- открывает товар;
- видит detail без модалки;
- видит similar;
- после similar продолжает исходную ленту;
- back возвращает в исходную позицию;
- данные приходят из нормализованного фида;
- товары без фото и недоступные товары не показываются;
- API поддерживает cursor, anchor и feed version.
